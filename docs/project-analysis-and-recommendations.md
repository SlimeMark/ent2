## What the Project Does

A face-tracking system running entirely on-device:

1. `CameraManager` — captures QVGA (320×240) frames from the built-in camera using `CoreS3.Camera.*`
2. `FaceDetector` — runs a 2-stage on-device ML pipeline (`HumanFaceDetectMSR01` → `HumanFaceDetectMNP01` from ESP-DL) to locate faces
3. `ServoController` — drives two continuous-rotation servos via PWM with a PID controller to center the face horizontally
4. `DisplayManager` — renders the live camera feed plus a bounding box overlay and FPS counter using M5GFX

The main loop throttles display at ~50 fps and face detection at ~8 fps. Servo updates are non-blocking using a pulse-timing state machine.

**Overall assessment:** The modular structure is clean and the code quality is solid for a prototype. Several latent bugs are likely preventing the system from running correctly in its current state. See below.

## Findings

### Critical — Likely Breaking

#### 1. Camera pixel format not set to RGB565 (`CameraManager.cpp:26-31`)

The face detector unconditionally returns "no face" if the frame format is not `PIXFORMAT_RGB565` (`FaceDetector.cpp:47-49`). However `CameraManager::init()` never sets the pixel format — the ESP32 camera driver defaults to **JPEG** unless explicitly configured.

```cpp
// CameraManager.cpp — current init()
bool CameraManager::init() {
    if (!CoreS3.Camera.begin()) return false;
    CoreS3.Camera.sensor->set_framesize(CoreS3.Camera.sensor, FRAMESIZE_QVGA);
    return true;  // pixel format is still JPEG — detector will always return no-face
}
```

**Fix:** Add the pixel format line immediately after `set_framesize`:

```cpp
CoreS3.Camera.sensor->set_pixformat(CoreS3.Camera.sensor, PIXFORMAT_RGB565);
```

---

#### 2. M5Unified and M5CoreS3 both declared as dependencies (`platformio.ini`)

`lib_deps` includes both `M5Unified` and `m5stack/M5CoreS3`. These libraries define overlapping symbols. In `main.cpp`, `M5.config()` (M5Unified API) is passed to `CoreS3.begin()` (M5CoreS3 API). In `DisplayManager.cpp`, `M5.Lcd` (M5CoreS3) is used. This mixed usage is fragile and can cause linker conflicts or undefined behaviour depending on initialization order.

**Fix for prototype:** Choose one library and be consistent.

- **Recommended path: M5Unified only.** Remove `m5stack/M5CoreS3` from `lib_deps`. Replace `CoreS3.begin(cfg)` in `main.cpp` with `M5.begin(cfg)`. Replace `CoreS3.Camera.*` with the M5Unified camera API, or use `esp_camera_*` directly. Replace `M5.Lcd.*` in `DisplayManager` with `M5.Display.*`.

- **If M5CoreS3 is kept:** Remove `M5Unified` from `lib_deps` and replace `M5.config()` with the CoreS3-native init pattern. Keep `M5.Lcd.*` as is.

> **Context for AI agents:** `M5.Lcd` is a macro/alias provided by `m5stack/M5CoreS3` pointing to the GFX display object. `M5.Display` is the M5Unified equivalent. They are functionally identical for drawing but are initialised differently. Using both in the same build typically compiles but may initialise the display twice, corrupting the SPI bus state.

### Medium — Functional Bugs

#### 3. Dead `state` parameter in `drawOverlay` (`DisplayManager.cpp:136-154`)

`drawOverlay(const char* state, int errorPx)` casts `state` to void and ignores it. The function actually displays FPS, not an error value, and the `errorPx` parameter is repurposed as the FPS integer. The call sites in `main.cpp` pass state strings like `"Centered"` and `"Adjusting"` that are silently discarded.

```cpp
// main.cpp:152-155 — state string is dropped on the floor
g_servoController.stopBoth();
g_display.drawOverlay("Centered", errorPx);   // "Centered" never shown
```

**Fix:** Either show the state string in the overlay, or rename `errorPx` to `fps` and remove the `state` parameter. For a prototype, showing both FPS and tracking state is useful debug info.

---

#### 4. Static local variable in `calculatePID` (`ServoController.cpp:159-162`)

```cpp
static float filteredDerivative = 0;  // shared across all instances, never reset
```

A `static` local persists for the lifetime of the program and is shared across all `ServoController` instances. It is also never reset when the face is lost, so derivative state from a previous tracking session bleeds into the next one.

**Fix:** Promote to a member variable `m_filteredDerivative`, initialised to `0` in the constructor. Also reset `m_pidIntegral` and `m_filteredDerivative` to 0 in `stopBoth()` or when face-lost is detected in `main.cpp`, to prevent stale PID state affecting the first move after reacquisition.

---

#### 5. PID integral accumulates without time normalization (`ServoController.cpp:148`)

`m_pidIntegral += m_pidError` accumulates on every `adjust()` call. But `adjust()` is only called when a face is detected, which happens at a variable rate (detection throttled, frames dropped, face intermittent). The integral therefore accumulates at an inconsistent rate and is not proportional to time-in-error. Under poor lighting where detection drops to 2–3 fps, the integral winds up far more slowly than under good conditions.

**Fix for prototype:** Track the timestamp of the last `adjust()` call and scale the integral contribution by elapsed time:

```cpp
uint32_t now = millis();
float dt = (now - m_lastAdjustMs) / 1000.0f;
m_pidIntegral += m_pidError * dt;
m_lastAdjustMs = now;
```

Add `uint32_t m_lastAdjustMs` as a member and initialise to `0`.

---

#### 6. Board definition mismatch (`platformio.ini`)

`board = esp32-s3-devkitc-1` is a generic DevKit board. PlatformIO has an `m5stack-cores3` board definition that sets the correct flash size (16 MB), PSRAM type (OPI PSRAM), and USB CDC mode for the CoreS3. Using the wrong board definition may affect flash partitioning and PSRAM access.

**Fix:**
```ini
board = m5stack-cores3
```

If this board isn't available in your PlatformIO version, add the board_build overrides manually:
```ini
board_build.flash_size = 16MB
board_build.partitions = default_16MB.csv
board_build.arduino.memory_type = qio_opi
```

### Low — Code Quality / Prototype Debt

#### 7. Two-pass iteration to find largest face (`FaceDetector.cpp:89-120`)

The results list is iterated twice: once to find `bestFaceIndex`, once to extract that face's data. A single pass with a running-best is cleaner and avoids re-iterating:

```cpp
const dl::detect::result_t* best = nullptr;
int maxArea = 0;
for (const auto& face : results) {
    if (face.box.size() >= 4) {
        int area = (face.box[2]-face.box[0]) * (face.box[3]-face.box[1]);
        if (area > maxArea) { maxArea = area; best = &face; }
    }
}
if (best) { /* extract faceInfo from best */ }
```

---

#### 8. Tilt axis unused — only horizontal (pan) tracking implemented

`faceInfo.centerY` is computed and stored but never used in `ServoController::adjust()`. The servo control is pan-only. This is presumably intentional for a 1-axis prototype, but worth noting explicitly so the Y-axis isn't forgotten when a tilt servo is added.

When adding tilt: the two servos currently receive identical pulse widths (`leftUs == rightUs` always). If the rig uses one servo for pan and one for tilt, the `adjust()` logic will need to be split.

---

#### 9. Face confidence score not used as a filter

`faceInfo.confidence` is populated from `face.score` but never checked. The detector can return low-confidence candidates, especially with partial faces or faces at the edge of frame. Adding a minimum confidence threshold (e.g., `0.6f`) avoids chasing shadows:

```cpp
// In FaceDetector::detectFace, before returning faceInfo
if (faceInfo.confidence < MIN_CONFIDENCE) {
    faceInfo = { -1, -1, 0, 0, 0.0f };
}
```

Add `static constexpr float MIN_CONFIDENCE = 0.6f;` to `FaceDetector.h`.

---

#### 10. `drawStatus` does a full `fillScreen` (`DisplayManager.cpp:47`)

`fillScreen` on the CoreS3 clears the entire 320×240 display over SPI, which takes several milliseconds. `drawStatus` is only called in init and error states (not in the main loop), so performance is not an issue now. However if this is ever adapted for runtime status updates, it will cause visible flickering. Note for future reference.

---

#### 11. All 4 PWM timers allocated for 2 servos (`ServoController.cpp:65-69`)

```cpp
ESP32PWM::allocateTimer(0);
ESP32PWM::allocateTimer(1);
ESP32PWM::allocateTimer(2);
ESP32PWM::allocateTimer(3);
```

Only 2 timers are needed for 2 servos. Allocating all 4 prevents any other PWM peripheral (e.g., LED, buzzer) from using a timer. For a prototype this is harmless, but worth noting if peripherals are added.

## Recommended Next Steps (Prototype Order)

1. **Fix #1 first** (pixel format) and **#6** (board definition) — these are the most likely reasons the face tracking appears not to work at all. One-line changes each.

2. **Resolve #2** (library conflict) — pick M5Unified only. It is the actively maintained path and avoids the CoreS3-specific library falling behind.

3. **Fix #4** (static local → member variable) and reset PID state on face-lost. This will make tracking noticeably smoother.

4. **Fix #3** (overlay state display) — once you can see the tracking state on screen, tuning PID is much easier.

5. **Add confidence filter (#9)** if you see the turret jittering toward non-face objects.

6. **PID tuning** — the defaults `kp=0.5, ki=0.1, kd=0.2` were not tuned on hardware. For continuous-rotation servos with a nudge-pulse approach, `kd` is often better set to `0` first and added incrementally, as the derivative of quantised pixel-error tends to be noisy.

7. **Tilt axis (#8)** when a second servo axis is added to the rig.

## Architecture Note for Future Growth

The current design runs everything on a single Arduino loop (no FreeRTOS tasks). This works for the current feature set. If WiFi/BLE features are added (e.g., the audio streaming discussed in `iphone-audio-analysis.md`), the ML inference blocking the loop for 100–300ms will cause WiFi stack starvation and disconnects.

At that point, split into FreeRTOS tasks:
- **Core 0:** Camera capture + face detection (compute-heavy, WiFi-sensitive)
- **Core 1:** Display update + servo control (timing-sensitive)

Protect shared `FaceInfo` with a `SemaphoreHandle_t` mutex or a `portMUX_TYPE` spinlock. The existing module structure (4 separate classes) maps cleanly onto this split with minimal refactoring.
